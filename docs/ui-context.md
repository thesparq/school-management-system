# UI Context – Design Tokens & Conventions

## Color Tokens

All colors are defined as CSS custom properties via Tailwind CSS v4’s `@theme` directive. The palette uses a calm, trustworthy blue as the primary brand color and a warm amber as the secondary accent. Success and error colors follow standard educational conventions (green for correct/completed, red for incorrect/overdue). Surface neutrals provide consistent backgrounds and text.

## Typography

| Token | Font Stack | Usage |
|:---|:---|:---|
| `--font-sans` | `'Inter Variable', system-ui, -apple-system, Segoe UI, Roboto, sans-serif` | Body text, UI labels, buttons, forms, breadcrumbs, navigation |
| `--font-mono` | `'JetBrains Mono Variable', 'Fira Code', monospace` | Code snippets, technical content, data tables (numeric alignment) |
| `--font-display` | `'Inter Variable', system-ui, sans-serif` (with tighter letter-spacing) | Page headings, lesson titles, card headers |

All fonts loaded via `@fontsource-variable/inter` and `@fontsource-variable/jetbrains-mono`. Fallbacks are standard system fonts.

## Border Radius Scale

| Token | Value | Usage |
|:---|:---|:---|
| `--radius-xs` | `calc(0.625rem * 0.5)` (5px) | Tiny elements |
| `--radius-sm` | `calc(0.625rem * 0.6)` (6px) | Checkboxes, tags, badges, tiny buttons |
| `--radius-md` | `calc(0.625rem * 0.8)` (8px) | Default: buttons, inputs, cards, dropdowns |
| `--radius-lg` | `0.625rem` (10px) | Larger cards, modals, dialogs |
| `--radius-xl` | `calc(0.625rem * 1.4)` (14px) | Sidebar, wide containers |
| `--radius-full` | `9999px` | Pills, avatars, circular indicators |

Computed from base `--radius: 0.625rem`. All shadcn-svelte components map to these tokens.

## Dark Mode

Dark mode is activated by adding the `.dark` class to `<html>`. A `ThemeToggle` component (moon/sun icon button in the navbar) manages this:

- On first visit: checks `localStorage('theme')`, falls back to `prefers-color-scheme`
- On toggle: flips `.dark` class on `<html>`, persists preference to `localStorage`
- All existing `dark:` Tailwind variants work automatically — no separate color palette needed

Dark mode tokens invert the shadcn semantic colors (`--background`, `--foreground`, `--card`, etc.) while preserving the `@theme` color scales (`primary-*`, `surface-*`, etc.) for component use.

## Component Conventions

### PageHeader

All pages use the shared `<PageHeader>` component for consistent heading + action layout:

```
<PageHeader
  title="Student Users"
  createLabel="Create Student"
  onCreate={() => showCreateDialog = true}
  backHref="/my-classes"     ← optional back arrow
/>
```

Renders: `<div class="flex items-center justify-between">` with optional back arrow, `<h1 class="text-2xl font-display font-bold text-primary-700 dark:text-primary-300">`, and optional `<AppButton>`.

### StatusCard

Use `StatusCard` for all persistent page-level states. Never use raw `<Alert>` or custom inline HTML for these:

| State | Component |
|-------|-----------|
| Error loading data | `<StatusCard variant="error" title="Failed to load..." description={error} onRetry={...} />` |
| Empty (no data yet) | `<StatusCard variant="info" title="No students yet" description="Create the first student..." />` |
| Warning (e.g., no active term) | `<StatusCard variant="warning" title="No active session term" description="Set one in Configuration." />` |
| Not initialized | `<StatusCard variant="info" title="Account not initialized" description="Contact your administrator." />` |

StatusCard is imported from `$lib/components/ui/status-card/status-card.svelte`.

### Toast Notifications

Use for **transient operation feedback** only (create/edit/delete/toggle success or failure). Never use for page-level error states.

```typescript
import { addToast } from '$lib/stores/toast';
addToast('success', 'Student created', username);
addToast('error', 'Save failed', err.message);
addToast('info', 'User activated', '');
```

Four variants: `success`, `info`, `warning`, `error`. Auto-dismiss 5s. Pause on hover.

### AppButton

Wrap shadcn `Button` with `loading` prop for all async operations. All form submit buttons use:

```svelte
<AppButton onclick={handleSave} loading={saving}>Save</AppButton>
```

Shows spinner + disables while `loading=true`. All other props passthrough.

## Page Layout Conventions

### Container Width

All list/table pages (users, session terms, terms, subjects, classes): `space-y-6` (full width).

Lesson detail pages: `mx-auto w-full max-w-5xl px-4 sm:px-6 lg:px-8` (centered, constrained).

### Form Dialogs

All create/edit dialogs use `Dialog` from shadcn-svelte with:
- `sm:max-w-lg` on `DialogContent`
- `DialogHeader` with `DialogTitle` and `DialogDescription`
- Password fields: show/hide toggle + generate button via `AppButton`
- Name parts: suffixed `(`*`)` on required labels
- Error display: `<p class="text-sm text-error-500">` inline
- Actions: Cancel (`AppButton variant="outline"`) + Save/Create (`AppButton loading={...}`)

### Forms

- All inputs use `Input` from `$lib/components/ui/input`
- Labels use `Label` from `$lib/components/ui/label`
- Password fields have show/hide toggle + generate button in `flex gap-2`
- Name parts use `NameFields.svelte` (surname, first_name, middle_name in responsive grid)
- Passport upload uses `PassportUpload.svelte` (drag-drop, preview, presigned R2 upload)

### Data Tables

All admin user tables follow the same column structure per role:

| Role | Columns |
|------|---------|
| Students | Name, Email, Class, Auth Status, Activate, Edit, Delete |
| Teachers | Name, Email, Auth Status, Activate, Edit, Assign, Delete |
| Admins | Name, Email, Role Title, Auth Status, Activate, Edit, Delete |
| Parents | Name, Email, Students, Auth Status, Activate, Edit, Delete |

Table components handle their own loading (skeleton rows), error (StatusCard), empty (StatusCard), and data states internally.

### Breadcrumbs

Rendered in `+layout.svelte` from `$page.data.breadcrumbs` array. Every server load function returns breadcrumbs for its route. Falls back to `[{ label: 'Dashboard' }]`.

### Card Grids

Responsive grid: `grid grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 xl:grid-cols-4 gap-4`.

Skeleton loading: 4-6 skeleton cards matching the grid layout during `$navigating` transitions.

## Passport Upload UX

### Component: `PassportUpload.svelte`

States:
- **Empty:** dashed border drop zone with "Click or drag passport photo" + "JPEG or PNG, max 5 MB" label
- **Preview:** 200×200 thumbnail with red "×" remove button
- **Uploading:** spinner overlay on thumbnail
- **Error:** red border + inline error message

Behavior:
- Client-side validates JPEG/PNG type and ≤5MB size on file select
- Does NOT upload on select — parent form's submit handler calls `getPassportPublicUrl()`
- Exposes two exported functions: `getPassportPublicUrl(file, profileType, userId)` and `getContentType()`

### Upload Flow

```
1. User selects file → client validates → preview shown
2. On form submit:
   a. POST /api/admin/generate-passport-upload-url { profileType, userId, contentType, fileSize }
   b. PUT file directly to presigned R2 URL
   c. Include returned publicUrl in create/edit API call
3. On success: toast, close dialog
4. On create failure: SvelteKit API route calls deleteR2Object() (saga compensation)
```

### Passport in Edit Mode

Pre-loads existing `passport` URL from profile and shows thumbnail. User can replace by selecting new file or keep existing.

## AI & Accent Color Variants

No special AI accent color is required for the MVP. The primary blue is sufficient for AI-related UI elements (e.g., an AI sparkle icon can remain blue). If a distinct "AI" identity is needed later, introduce a **purple/violet** accent (`#8b5cf6` – `#7c3aed`) reserved only for AI-generated content labels, badges, and subtle glows; it must never replace the semantic colors for success or error.

## Implementation Note (Tailwind v4)

In `app.css`, define the tokens like this:

```css
@import "tailwindcss";

@theme {
  --color-primary-50: #eff6ff;
  --color-primary-100: #dbeafe;
  /* … all other tokens … */
  --font-sans: 'Inter Variable', system-ui, sans-serif;
  --radius-md: 0.375rem;
}
```

Then use them as `bg-primary-500`, `text-surface-800`, `rounded-md`, etc. The design system stays consistent and fully under version control.
