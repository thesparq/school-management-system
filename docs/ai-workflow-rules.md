# AI Workflow Rules

These are instructions for any AI coding agent working on this project. Follow them strictly. They are rules, not suggestions.

## Overall Approach

- **Follow the project context files as the source of truth.** Before writing any code, read `project-overview.md`, `architecture.md`, `code-standards.md`, and `ui-context.md`. Every line of code must align with the architecture and invariants defined there.
- **Work spec-first.** Every unit of work begins with a clear specification of what changes, why, and where. Do not write code before the specification is agreed upon.
- **Deliver incrementally.** Break work into the smallest possible units that can be independently verified. Each unit must produce a buildable, testable state. Never batch unrelated changes into one step.
- **End each unit at a clean boundary.** After completing a unit, the project must build without errors, all existing functionality must remain intact, and no broken intermediate state may be left behind.

## Scoping Rules

- **Work on one unit at a time.** A unit is a single, well-defined change: one agent method, one UI route, one data migration, one WIT interface addition. Do not drift into adjacent concerns.
- **Do not make speculative changes.** If a change is not required by the current unit, do not make it. Do not "clean up" unrelated code. Do not refactor unless the unit explicitly requires it.
- **Do not add features not requested.** If the specification defines a method with three parameters, implement exactly three parameters. Do not add optional fields or extra functionality you anticipate might be useful later.
- **If you see a problem outside the current scope, report it.** Say: "I noticed X might cause Y. Should I address this now or file it for later?" Do not act on it without explicit instruction.

## Splitting Work

Break a unit into smaller units when any of the following are true:

- The change spans more than two system boundaries (e.g., SvelteKit route + agent RPC + database query).
- The change affects more than one agent component or more than one SvelteKit route group.
- The implementation requires more than three files to be created or modified.
- The specification for the unit is longer than half a page.
- Any part of the unit has an unclear requirement that blocks the rest.

When splitting, present the sub-units as a numbered list and request approval before proceeding. Implement each sub-unit as an independent step that builds and passes verification.

## Handling Missing or Ambiguous Requirements

- **Stop immediately.** Do not guess. Do not assume a reasonable default. Do not proceed with incomplete information.
- **State exactly what is missing.** Say: "The specification does not define X. Options are: (a) ..., (b) ..., or (c) .... Which should I use?"
- **Do not invent requirements.** If a business rule is not stated (e.g., "what happens when a student submits after the deadline?"), do not implement any behaviour. Flag it and wait for a decision.
- **If the ambiguity blocks only part of the unit,** isolate the blocked portion, implement the rest, and clearly mark the blocked part with a comment containing `TODO(SPEC)` and the open question.

## Protected Files

The following must never be modified without explicit, direct instruction in the current unit:

- Any file inside `frontend/src/lib/components/` that was generated by `shadcn-svelte` (the `npx shadcn-svelte add` output). These are library code; compose them, do not edit them.
- `moon.work` and any `moon.mod.json` or `moon.pkg.json` files. Workspace and package configuration changes require explicit approval.
- `golem.yaml` files. Infrastructure and deployment manifests are changed only when a unit explicitly requires it and after confirmation.
- `vite.config.ts`, `svelte.config.js`, `tsconfig.json`, and `package.json`. Build tooling configuration is changed only with explicit instruction.
- Any `.wit` file that has already been deployed or is shared by multiple agents. Changing a WIT interface may break RPC contracts across agents.

## Documentation Synchronization

- **Documentation and implementation are a single unit.** When a unit adds or changes behaviour, update the relevant context file in the same step.
- **Which file to update for which change:**
  - New or changed agent responsibilities → `architecture.md`
  - New or changed data schemas, storage decisions → `architecture.md` (Storage Model section)
  - New invariants or rule changes → `architecture.md` (Invariants section)
  - New UI patterns, component conventions → `ui-context.md`
  - New coding rules, file organization changes → `code-standards.md`
  - New feature scope, user flow changes → `project-overview.md`
  - New feature, endpoint, or user-facing capability → `feature-checklist.md` (add a test row)
- **If a change invalidates a statement in a context file, you must correct it.** Leaving stale documentation is a bug.
- **Keep documentation concrete.** Replace vague phrases with specific names, counts, and paths. If the implementation uses `assignment_id`, the documentation must use the same term, not "assignment reference" or "assignment key."

## Verification Checklist

Before marking a unit as complete, verify every item on this list. If any item fails, fix it before proceeding.

1. **Build check:** Run the build command for the affected project (`moon build` for `shared/` and `agents/`, `pnpm check && pnpm build` for `frontend/`). The build must succeed with no errors and no new warnings.
2. **No regressions:** All previously working routes, agent methods, and RPC calls remain functional. If a change could affect existing behaviour, explain why it does not.
3. **Invariants preserved:** Review the invariants section in `architecture.md`. Confirm the change does not violate any invariant. If it does, stop and raise the conflict.
4. **Documentation updated:** The relevant context file reflects the change. No stale or contradictory statements remain.
5. **Code standards met:** The change follows all rules in `code-standards.md`. Check types, validation, error handling, naming, and file placement.
6. **Clean boundary:** The change does not leave partially implemented methods, commented-out code, TODO markers (unless explicitly approved), or debug logging.
7. **Auth and access verified:** If the change touches an API route or agent method, confirm that auth checks (JWT validation, role check, ownership check) are present and cannot be bypassed.
8. **State migration verified:** If the change modifies an agent's durable state (struct fields), confirm that new fields have safe defaults and existing agents are compatible with the new struct shape.
9. **Explicit sign-off:** State: "Unit complete. Built successfully. Verified items: [list]. Ready for next unit."
